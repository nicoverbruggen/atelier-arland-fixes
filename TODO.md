## Atelier Rorona

- Investigate a cut-in shadow correctness glitch (also occurs in Meruru — the cut-in shadow code is shared between the two, so fix once). The restored shadows themselves are correct, but during the action cut-ins the engine juggles the non-focus characters around: sometimes it hides them, and sometimes it moves them off-position (they appear above the cut-in scene). The restored shadow reception still projects those casters onto the ground, so ground shadows appear for characters that are invisible or not where the shadow is. The problem is a caster shadowing the floor while its character has been repositioned or hidden for the close-up — likely the same root cause as the Meruru pre-victory-screen artifact below. Worth a thorough investigation: determine how the engine repositions/hides the non-focus battlers during cut-ins, and gate the restored shadow reception (or the individual caster's contribution) so only actually-visible, on-position characters cast cut-in shadows. Until this is fixed, cut-in shadows and cut-in brightening are OFF by default (`CutInShadows`/`CutInDimHold` are OptIn in the capability matrix for Rorona and Meruru); flip both back to `OnByDefault` in `game.cpp` once the glitch is stable.
- Check whether the animated-portrait (BUC) conversation slowdown also occurs in Rorona. The same `Contrast` + global mechanism is structurally present (RVAs not yet pinned), but the slowdown has not been reproduced in Rorona yet.

## Atelier Meruru

- Validate the animated-portrait (BUC) conversation slowdown fix. The slowdown was observed only during field-map conversations (`nspFM` talk/message/event), ending the instant the conversation does; cause: the balloon re-runs the EN slow text-render path (`renderText 0x5115d0` → atlas `Lock 0x3ea7d0`) every frame via its per-frame callback pump. The implemented fix hooks the `BalloonBucMode` ctor/dtor (`0x1e8c40`/`0x1e8d30`, English build) and, while any balloon is alive, activates the text-bitmap replay cache with a cross-frame scope so unchanged text is replayed as a memcpy. Validate on the deterministic repro: the first Lias event after the first combat ("Whew, that one gave me a hard time" — event `QC02_LIA`) — the field framerate should stay smooth for the whole conversation. Also confirm menus have not regressed (the fix shares the menu-hitch cache machinery). `ARLAND_MENU_STATS=1` logs a `BUC balloon scope end hits=… misses=…` line per conversation (expect hits ≫ misses); `ARLAND_BUC_TEXT_CACHE=0` disables the fix for A/B comparison. The multilingual build's `BalloonBucMode` ctor/dtor RVAs (`0x1d97a0`/`0x1d9890`) were homologue-matched and the fix is installed for both builds; whether the slowdown reproduces on the Japanese executable — and that the fix behaves there — is untested (runtime validation pending).
- Validate the battle-end watchdog fix for the post-combat field slowdown. Observed (heartbeat log, 2026-07-23): after a battle ends and play returns to the already-loaded field, the mod's battle tracking never disengaged — no `BATTLE_END`, `battle_active=1` and a stuck cinematic state flag scanned from the freed battle game-mode persisted through field exploration (text rendering was measured and ruled out as the cost; frame drops in that window coincided with the stuck state). Cause: battle exit was only detected via the field `ShadowHelperInit` call site, which does not run when returning to an already-loaded field, and the existing game-mode-dead watchdog only armed when a helper had been published (Rorona-only). Fix: the watchdog now arms whenever battle tracking is active, after the game-mode has been seen alive once, and fully clears the battle-tracking state (`BATTLE_END` log line). Validate: repeat field → combat → field; the log should show `BATTLE_END` shortly after the battle result, `battle_active=0`/`cinematic=0` heartbeats on the field, and no post-combat frame drops.
- Fix a cut-in shadow artifact just before the victory screen: for a few frames, shadows are visible with the characters apparently positioned up in the sky. This is very likely a specific instance of the cut-in character-juggling shadow glitch (see the Atelier Rorona section) — non-focus characters repositioned above the scene while their casters still shadow the floor. Investigate the cause and suppress it during the victory/result transition. (This is the "verify victory screens" item.)
