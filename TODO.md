- Supersampling factor: a new `[Rendering]` field (e.g. `SupersamplingFactor`) independent of `Width`/`Height` — render the scene internally at `factor ×` the output resolution and downscale (SSAA). Feasibility investigation complete (2026-07-23), verdict viable. Chosen architecture: "big internal, small backbuffer" — reuse the existing resize/viewport machinery unchanged at `output × factor` (all fixed-size assumptions live on the internal side) and keep the swap chain at output size, so the only new work is the single scene→backbuffer boundary. Key findings: the backbuffer must be classified **by identity** (private-data tag at swap-chain creation), because at factor 2 `internal/2 == output` and size-based classification collides with the half-res blur targets; whether the boundary is a fullscreen draw (SRV substitution, bilinear is exact for factor 2) or a copy (needs a mod blit) requires a half-day runtime probe first; UI renders supersampled (same as today's 4K-native path — least breakage); MSAA should be mutually exclusive in the first iteration (4×MSAA on 2×SSAA at 1440p ≈ 450 MB for one target pair). VRAM delta at 1440p output: ≈ +300 MB at 2×, ≈ +1.4 GB at 4×; GPU cost scales ~factor². Plan: probe → factor-2 core → downscale pass (enables 3×/4×) → clamps and docs; ≈ 1–1.5 weeks total. Independent quick win: document the gamescope recipe (`Width/Height=2×` + gamescope downscale) in ADVANCED.md — it gives Linux users SSAA with the mod as-is today.
- Add SMAA (post-process anti-aliasing) as a rendering option. Precedent confirmed (2026-07-23): Atelier Graphics Tweak shipped SMAA for these games (its archive contains `SMAA.hlsl`/`SMAAWrapper.hlsl` and the DLL carries the full DX11 pipeline at `SMAA_PRESET_ULTRA`, applied full-frame per resolution) — AGT has no SSAA; SMAA is what it offered. SMAA is one cheap full-frame pass (edge detection → blend weights → neighborhood blending), smooths thin-trim/alpha edges that MSAA misses, and composes with the mod's other features. The reference implementation (Jimenez et al.) is permissively licensed; the mod already has the prerequisites (runtime shader compilation, and either the pre-UI injection transition mapped by the SSAO investigation or AGT's simpler full-frame-at-present placement, which slightly smooths UI). Candidate `[Rendering]` key (e.g. `SMAA=true`); decide placement (pre-UI vs full-frame) when implementing.

- Investigate per-sample (sample-rate) shading under MSAA: rewrite pixel shaders to execute per sample so shader-interior aliasing (not just geometric edges) is multisampled. Prior art to lean on: TellowKrinkle's atelier-sync-fix rendering branch implements the full DXBC transform (ShaderSampleRateConverter). - Fix the jagged aliasing on thin costume trim (frilled lace on sleeves/collars, hair fringe; observed on Meruru's dress, mechanism engine-wide). Investigation complete (2026-07-23), answered statically by parsing the PSSG model archives: every frill/lace material enables BOTH alpha blending (`SRC_ALPHA/INV_SRC_ALPHA`) AND a shader-side alpha test (`discard` against `g_alpha_ref`, threshold 0.1) — confirmed in Rorona/Totori/Meruru player models. The trim textures are DXT1 with 1-bit alpha, so the blendable gradient is a sub-pixel bilinear band and the visible edge is a hard cutout — classic alpha-test aliasing that MSAA cannot touch (texture-interior, not a silhouette). **This is the alpha-to-coverage case.** Plan: (1) half-day runtime probe (`CreatePixelShader` DXBC scan for discard/`g_alpha_ref` to classify character shaders, `CreateBlendState`/`CreateSamplerState` logging, draw-time classification); (2) draw-time A2C substitution — when MSAA is active and a classified alpha-test shader draws into the MSAA twin, bind a cached clone of the current blend state with `AlphaToCoverageEnable`, restore after (never mutate the game's shared state objects — creation-time patching would dither every alpha-blended draw in the game); ini key e.g. `TrimAlphaToCoverage`, meaningful only with MSAA. Caveats: DXT1's binary alpha caps the smoothness ceiling; A2C+blend double-applies alpha (ship a BlendEnable-off A/B variant); hair fringe shares the states and will change too. Optional complement: anisotropic substitution on classified character draws only (no LOD bias — negative bias adds shimmer on binary-alpha textures). ~1–2 days after the probe. Relates to the anisotropic-filtering and SMAA items above. Prior art to lean on: TellowKrinkle's atelier-sync-fix rendering branch ships an alpha-to-coverage path via shader replacement (custom Shadows/Environment HLSL variants), proving the visual approach on this engine; our blend-state substitution plan is the lighter alternative.
- Add SSAO (screen-space ambient occlusion). Feasibility investigation complete (2026-07-23): practical with the mod's existing infrastructure. Key findings: identify the main depth buffer via the existing creation/bind tracking and read it through a mod-owned typeless "depth twin" copy (no format promotion of engine resources); reconstruct normals from depth derivatives (no G-buffer exists; the blob machinery already captures the projection constants needed for linearization); inject the fullscreen AO pass at the already-tracked "main RT+depth pair unbound" transition, which lands before UI composition (injecting at Present would darken menus); the pass records on the deferred context and must save/restore pipeline state. Conflict: under MSAA the only populated depth is the multisampled twin, so the first iteration should disable SSAO when MSAA is active. Plan: (1) probe run — log the main depth desc and the per-game UI-boundary transition; (2) minimal native SSAO (half-res AO + blur + multiply composite, `SSAO` ini key, opt-in); (3) quality pass (depth-aware blur, per-game tuning); (4) optional MSAA support. Alternative documented option: ReShade chain-loading through the existing `d3d11_proxy.dll` load path (free MXAO, but AO darkens UI since ReShade runs at Present).
