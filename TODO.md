- Supersampling factor: a new `[Rendering]` field (e.g. `SupersamplingFactor`) independent of `Width`/`Height` — render the scene internally at `factor ×` the output resolution and downscale (SSAA). Feasibility investigation complete (2026-07-23), verdict viable. Chosen architecture: "big internal, small backbuffer" — reuse the existing resize/viewport machinery unchanged at `output × factor` (all fixed-size assumptions live on the internal side) and keep the swap chain at output size, so the only new work is the single scene→backbuffer boundary. Key findings: the backbuffer must be classified **by identity** (private-data tag at swap-chain creation), because at factor 2 `internal/2 == output` and size-based classification collides with the half-res blur targets; whether the boundary is a fullscreen draw (SRV substitution, bilinear is exact for factor 2) or a copy (needs a mod blit) requires a half-day runtime probe first; UI renders supersampled (same as today's 4K-native path — least breakage); MSAA should be mutually exclusive in the first iteration (4×MSAA on 2×SSAA at 1440p ≈ 450 MB for one target pair). VRAM delta at 1440p output: ≈ +300 MB at 2×, ≈ +1.4 GB at 4×; GPU cost scales ~factor². Plan: probe → factor-2 core → downscale pass (enables 3×/4×) → clamps and docs; ≈ 1–1.5 weeks total. This is the definitive fix for the fine costume-trim aliasing (see below) — no cheaper technique resolves sub-pixel alpha-test lace. Independent quick win: document the gamescope recipe (`Width/Height=2×` + gamescope downscale) in ADVANCED.md — it gives Linux users SSAA with the mod as-is today.
- Add SMAA (post-process anti-aliasing) as a rendering option. Precedent confirmed (2026-07-23): Atelier Graphics Tweak shipped SMAA for these games (its archive contains `SMAA.hlsl`/`SMAAWrapper.hlsl` and the DLL carries the full DX11 pipeline at `SMAA_PRESET_ULTRA`, applied full-frame per resolution) — AGT has no SSAA; SMAA is what it offered. SMAA is one cheap full-frame pass (edge detection → blend weights → neighborhood blending), smooths thin-trim/alpha edges that MSAA misses, and composes with the mod's other features. The reference implementation (Jimenez et al.) is permissively licensed; the mod already has the prerequisites (runtime shader compilation, and either the pre-UI injection transition mapped by the SSAO investigation or AGT's simpler full-frame-at-present placement, which slightly smooths UI). Candidate `[Rendering]` key (e.g. `SMAA=true`); decide placement (pre-UI vs full-frame) when implementing.
- High-resolution UI text. Feasibility probe complete (2026-07-23): the UI font is NOT scalable — it is a pre-baked Koei Tecmo **G1N bitmap font** (`Res/x64/font/ArlandDX_font_jp.g1n`, 28,831 glyphs all 32×48px 4bpp; `_ch.g1n` 33,422 glyphs 32×32), blitted **1:1 with no scaling** into 512×512 atlases; GDI32 imports only `GetStockObject` (no CreateFont/DirectWrite/FreeType). So there is no "render bigger" input and no higher-res glyph source in the game. (The "sharp Totori main-menu font" lead was chased 2026-07-23 and CLOSED: that text is pre-baked high-res texture art — sprites cut from `a12_title_text.g1t` 1024×512 BC3 in `a12_title_EN.PSSG`, drawn via a textured-quad sprite path that never touches the glyph renderer — not a scalable font. All three games bake their title menus this way; there is no scalable/higher-res glyph source anywhere.) HOWEVER the bitmap-swap idea (the user's) IS viable: the text pipeline's UV/quad math is **fully normalized end-to-end** (quad geometry gets used-fraction UVs + aspect only, never a raw pixel dim), so substituting a 2×/4×-content string bitmap keeps the same on-screen quad at higher texel density. The mod already owns the substitution hook (`cachedRenderText`, menu_fix.cpp). Real added detail must come from an EXTERNAL source: **B1** bundle/user-supply a lookalike TTF (Tuffy is public-domain, close for Latin; EN-first, no shipped CJK vector) and re-render glyphs at 2×/4× via FreeType/DWrite — best quality; **B2** offline-vectorize the G1N glyphs → re-render at 64×96 (exact typeface, covers JP/CH, "derived" detail, needs a generator tool run on the user's own assets). Both then feed the hi-res bitmap through cachedRenderText, replicating the game's per-char advance (call 0x55ecb0), 5/8 EN row-crop, pow2 dims, and fraction metrics. Recommended de-risk before committing: force-double one cached bitmap nearest-neighbor and confirm unchanged on-screen size/pos (proves no hidden raw-dim consumer), and confirm all visible text routes through the renderText string-bitmap path (not a direct atlas-quad). Pipeline RVAs (Rorona EN): init 0x13bb40, renderText 0x5613b0, glyph-blit 0x4107a0, copy-out 0x561640, advance table 0x55ecb0, TextBlock consumer 0x562bd0. Full detail in the [[atelier-font-pipeline]] memory.

## Crash diagnosis and hardening (from Totori DX crash research, 2026-07-23)

Community crash research on Totori DX (Steam 936180) and its DX siblings established which crash classes our `d3d11.dll` proxy can act on. Full source list and analysis in that research; key actionable items:

- **Surface the faulting module in the crash post-mortem prominently.** The best-corroborated Totori crash is an in-battle crash preceded by an audio *screech* (triggered by bombs/crafts, worse in dense fights like Calamitous Mountain), which multiple users report can hardlock the PC. Signature + Valve's XAudio2 Proton tag point to the **audio path (XAudio2), off the D3D11 device** — we cannot fix it at this layer, but our crash logger is exactly what confirms it. Make sure the log makes the faulting `module+offset` unmissable (e.g. call out when it resolves to `xaudio2_*.dll` / `x3daudio*` vs the GPU driver vs the game). This resolves the biggest open question and, if confirmed audio-side, points at an XAudio2 shim as the real (separate) lever — not this DLL.
- **XAudio2 guard shim for the battle-screech crash (v0.6 candidate).** Scoping probe complete (2026-07-23). A drop-in proxy `xaudio2_N.dll` is NOT viable: none of the games import or `LoadLibrary` XAudio2 by name — all three create **XAudio2 2.7** via `ole32!CoCreateInstance` (`CLSID_XAudio2` {5a508685-…}, `IID_IXAudio2` 2.7, confirmed by the 2.7-only `Initialize(0, XAUDIO2_ANY_PROCESSOR)` call), which loads from an absolute system path and never checks the game dir. The viable path is a **MinHook detour chained from the existing `d3d11.dll` proxy** (it loads before the sound device is created): (1) hook the game's sound-device init — byte-identical across all three EN builds, prologue `40 55 56 57 41 54 41 55 41 56 41 57 48 8d ac 24`, RVAs rorona-en 0x4176e0 / totori-en 0x4fc1c0 / meruru-en 0x403840 (ML builds via homolog) — and read `IXAudio2*` from `[device+0x1c8]` after calling the original; (2) MinHook `IXAudio2::CreateSourceVoice` (2.7 vtable slot 8, +0x40), and in it one-time-hook the returned voice's `SubmitSourceBuffer` (source-voice vtable slot 21, +0xA8; all voices share one vtable) and wrap the caller's `IXAudio2VoiceCallback*` in an SEH-guard object. **Asset audit result (2026-07-23): the audio ASSETS are exonerated.** All 262 Totori PCM SE descriptors (banks `BASIC01/BASIC02/PSSG_OBJ01.whd/.wbd`, `_HBW0000`/`KWB2` raw PCM16) + the KOVS/Vorbis `.fp` streams were scanned two ways (marker sweep + authoritative offset-table walk) — ZERO anomalies (no OOB offsets, no zero-length, no bad sample rate/channels, no codec issues) and structurally indistinguishable from the non-crashing Rorona/Meruru banks (Meruru even has MORE empty-slot irregularities and doesn't crash). The SE carry **NO loop metadata** (one-shots), so a "LoopBegin+LoopLength > AudioBytes" screech is structurally impossible from the assets — any infinite-loop must be set by game code. The one Totori-unique asset (`sta.fp`, loaded first in the `.fp` list) is 4 well-formed mono/48kHz Vorbis one-shots. Container/consumer RVAs found: sound-init/XAudio2-create 0x4fc1c0, KWB2 magic check 0x508da0, KOVS check 0x519f80, the exe's single `XAUDIO2_E_INVALID_CALL` (0x88960001) null-buffer guard at 0x4faa5c in func 0x4faa20. **Revised fix implication: a `SubmitSourceBuffer` field-clamp/loop-validation shim has NOTHING to clamp and will NOT fix this.** Combined with the RAM→100% hang reports (JRThought42, Kipper — memory leak, Scorched Wasteland bombs), the mechanism is a **consumer-side allocation/retry loop** (a failing `CreateSourceVoice`/`SubmitSourceBuffer` re-issued every frame), not an asset OOB. The right interception is the **SE-trigger / voice-acquire layer with drop-after-first-failure or per-SE rate-limiting** (so a failing create/submit can't be re-issued each frame and leak), NOT per-buffer validation. Copy-on-submit is still worth having for genuine use-after-free but is secondary. **Needs a runtime probe first:** instrument `CreateSourceVoice`/`SubmitSourceBuffer` HRESULTs + call frequency during a bomb throw to catch which SE index is retried/re-allocated in a tight loop (the exact voice wrapper isn't the naïve `call [rax+0x38]` and needs live pinning). Scan scripts saved in scratchpad (`scan_whd.py`, `walk_whd.py`). Note: this would be the mod's first hook into a non-game module (Wine's `xaudio2_7.dll`/FAudio builtin under Proton — mechanically fine, it's PE). **Gated on confirm-first:** do NOT build until a reproduced Totori bomb/craft screech-crash produces an `arland-fix.log` classifying the fault (or a stack frame) as `category=AUDIO` (expect `xaudio2_7.dll`). If the log shows GAME/GRAPHICS with no audio frames, the premise is wrong and this is dropped. Honesty caveats: init is byte-identical Rorona↔Totori so the bug is data/usage-dependent (bomb SE content), not an init difference; if the fault is deep in FAudio's mixer only copy-on-submit helps; a true audio-driver/PipeWire hardlock may be beyond any userspace shim (and may prevent the log flushing — a partial log with an audio-thread frame is still evidence).
- **Menu-close / battle-enter crash — ROOT CAUSE FOUND; proper fix pending confirmation.** The `VSync=false`/`SyncInterval=0` dodge was implemented then **REMOVED** (2026-07-23) at the user's request — they want the real fix, not a timing dodge. RETAINED as diagnostics: **device-lost logging** on Present (`Present device lost: result=… reason=…` via `GetDeviceRemovedReason`, logged once, main.cpp) and the periodic **`MEM` working-set line**.
