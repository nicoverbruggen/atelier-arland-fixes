## Crash diagnosis and hardening (from Totori DX crash research, 2026-07-23)

Community crash research on Totori DX (Steam 936180) and its DX siblings established which crash classes our `d3d11.dll` proxy can act on. Full source list and analysis in that research; key actionable items:

- **Surface the faulting module in the crash post-mortem prominently.** The best-corroborated Totori crash is an in-battle crash preceded by an audio *screech* (triggered by bombs/crafts, worse in dense fights like Calamitous Mountain), which multiple users report can hardlock the PC. Signature + Valve's XAudio2 Proton tag point to the **audio path (XAudio2), off the D3D11 device** — we cannot fix it at this layer, but our crash logger is exactly what confirms it. Make sure the log makes the faulting `module+offset` unmissable (e.g. call out when it resolves to `xaudio2_*.dll` / `x3daudio*` vs the GPU driver vs the game). This resolves the biggest open question and, if confirmed audio-side, points at an XAudio2 shim as the real (separate) lever — not this DLL.
- **Battle bomb/Craft crash (audio) — ROOT CAUSE CONFIRMED (2 Opus probes + Fable, 2026-07-23); fix specified, not built.** It is a reclaim-starvation MEMORY LEAK (not a screech/asset/retry-loop bug): the SE play-starter **`0x2b68c0`** (Totori EN; homolog-MATCH Meruru `0x3ce870`, prologue `40 53 48 83 ec 50 48 c7 44 24 20 fe ff ff ff 48`) unconditionally `new(0xc0)`s a play-instance and overwrites `[src+0x48]` (store `mov [rbx+0x48],r9` @`0x2b6905`) **WITHOUT freeing the prior one** → orphans the whole voice bundle (play-inst 0xc0 + voice-inst 0x50 + CSeVoice + XAudio2 source voice). ktsl reclaims a voice ONLY on `OnStreamEnd` (`0x509bc0`), while `OnBufferEnd` (`0x509b90`) is a bare `ret`, so bomb/Craft re-triggers in dense fights leak until RAM exhausts → hang. Fable-confirmed the overwrite store, the engine's own reclaim sequence @`0x2b58a2`, the play-inst dtor `0x25be0`, and the callbacks. Code byte-identical across the trilogy → Totori just hits the re-trigger rate. **FIX A (recommended, corruption-safe):** detour `0x2b68c0`; if `[rcx+0x48]` is non-NULL on entry, run the engine's OWN teardown on it — call dtor `0x25be0(old)` (Releases the CSeVoice → stops playback) then `operator delete 0x595828(old, 0xC0)`, then set `[rcx+0x48]=0` — before the original allocates. Caps to one live play-instance per source. Fix B (force-reclaim at create-voice `0x26100`) REJECTED (riskier — unpinned voice-container offset; DestroyVoice would dangle `[playinst+0xb0]`). **RUNTIME CONFIRM-FIRST (the smoking-gun A/B):** log `[rcx+0x48]` at `0x2b68c0` entry (non-NULL = an orphan about to be created) alongside the `MEM` line — under bomb spam the un-fixed build shows repeated orphans + a monotonic RAM climb; with Fix A the climb should stop. Alloc/free probe sites: alloc+ `0x2b68e6`/`0x26171`, create-voice `0x26100`, free− dtor `0x25be0` + delete `0x2b58bb`. NOT a drop-in `xaudio2_N.dll` (COM-loaded). Investigation history and rejected approaches (XAudio2 guard-shim scoping, asset audit, retry-loop hypothesis) below — a drop-in proxy `xaudio2_N.dll` is NOT viable: none of the games import or `LoadLibrary` XAudio2 by name — all three create **XAudio2 2.7** via `ole32!CoCreateInstance` (`CLSID_XAudio2` {5a508685-…}, `IID_IXAudio2` 2.7, confirmed by the 2.7-only `Initialize(0, XAUDIO2_ANY_PROCESSOR)` call), which loads from an absolute system path and never checks the game dir. The viable path is a **MinHook detour chained from the existing `d3d11.dll` proxy** (it loads before the sound device is created): (1) hook the game's sound-device init — byte-identical across all three EN builds, prologue `40 55 56 57 41 54 41 55 41 56 41 57 48 8d ac 24`, RVAs rorona-en 0x4176e0 / totori-en 0x4fc1c0 / meruru-en 0x403840 (ML builds via homolog) — and read `IXAudio2*` from `[device+0x1c8]` after calling the original; (2) MinHook `IXAudio2::CreateSourceVoice` (2.7 vtable slot 8, +0x40), and in it one-time-hook the returned voice's `SubmitSourceBuffer` (source-voice vtable slot 21, +0xA8; all voices share one vtable) and wrap the caller's `IXAudio2VoiceCallback*` in an SEH-guard object. **Asset audit result (2026-07-23): the audio ASSETS are exonerated.** All 262 Totori PCM SE descriptors (banks `BASIC01/BASIC02/PSSG_OBJ01.whd/.wbd`, `_HBW0000`/`KWB2` raw PCM16) + the KOVS/Vorbis `.fp` streams were scanned two ways (marker sweep + authoritative offset-table walk) — ZERO anomalies (no OOB offsets, no zero-length, no bad sample rate/channels, no codec issues) and structurally indistinguishable from the non-crashing Rorona/Meruru banks (Meruru even has MORE empty-slot irregularities and doesn't crash). The SE carry **NO loop metadata** (one-shots), so a "LoopBegin+LoopLength > AudioBytes" screech is structurally impossible from the assets — any infinite-loop must be set by game code. The one Totori-unique asset (`sta.fp`, loaded first in the `.fp` list) is 4 well-formed mono/48kHz Vorbis one-shots. Container/consumer RVAs found: sound-init/XAudio2-create 0x4fc1c0, KWB2 magic check 0x508da0, KOVS check 0x519f80, the exe's single `XAUDIO2_E_INVALID_CALL` (0x88960001) null-buffer guard at 0x4faa5c in func 0x4faa20. **Revised fix implication: a `SubmitSourceBuffer` field-clamp/loop-validation shim has NOTHING to clamp and will NOT fix this.** Combined with the RAM→100% hang reports (JRThought42, Kipper — memory leak, Scorched Wasteland bombs), the mechanism is a **consumer-side allocation/retry loop** (a failing `CreateSourceVoice`/`SubmitSourceBuffer` re-issued every frame), not an asset OOB. The right interception is the **SE-trigger / voice-acquire layer with drop-after-first-failure or per-SE rate-limiting** (so a failing create/submit can't be re-issued each frame and leak), NOT per-buffer validation. Copy-on-submit is still worth having for genuine use-after-free but is secondary. **Needs a runtime probe first:** instrument `CreateSourceVoice`/`SubmitSourceBuffer` HRESULTs + call frequency during a bomb throw to catch which SE index is retried/re-allocated in a tight loop (the exact voice wrapper isn't the naïve `call [rax+0x38]` and needs live pinning). Scan scripts saved in scratchpad (`scan_whd.py`, `walk_whd.py`). Note: this would be the mod's first hook into a non-game module (Wine's `xaudio2_7.dll`/FAudio builtin under Proton — mechanically fine, it's PE). **Gated on confirm-first:** do NOT build until a reproduced Totori bomb/craft screech-crash produces an `arland-fix.log` classifying the fault (or a stack frame) as `category=AUDIO` (expect `xaudio2_7.dll`). If the log shows GAME/GRAPHICS with no audio frames, the premise is wrong and this is dropped. Honesty caveats: init is byte-identical Rorona↔Totori so the bug is data/usage-dependent (bomb SE content), not an init difference; if the fault is deep in FAudio's mixer only copy-on-submit helps; a true audio-driver/PipeWire hardlock may be beyond any userspace shim (and may prevent the log flushing — a partial log with an audio-thread frame is still evidence).
